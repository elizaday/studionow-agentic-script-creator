# Client Script Output

This file defines the deliverable the client sees. Clean, presentable, production-ready on the page.

No producer notes. No asset sourcing. No approval flags. Those live in the Producer Notes document (07).

## Separation from producer notes

The client script and producer notes are separate deliverables. In the response, output them as two clearly separated sections:

1. Output the complete client script first
2. Then output a markdown horizontal rule: `---`
3. Then output `## Producer Notes` or `## PRODUCER NOTES` as a heading
4. Then output the complete producer notes

This separator allows downstream splitting. Do not blend producer-note language into the client script.

## Metadata header

Every script opens with:

```
[TITLE IN ALL CAPS]

Client: [Client name]
Writer: StudioNow Creative
Date: [M/D/YY]
Version: [Number]

Format: [Runtime + type, e.g., :90 Sizzle Reel]
Tone: [Selected tone from 08_tone_system]
Label: Internal Only | Confidential | External
```

Optional fields when relevant:
- Case Title
- Category
- Initiative
- Working Title

## Three-column table format

Use this structure:

| AUDIO / VO | TC | VISUALS |

- AUDIO / VO: voiceover lines, SFX cues, music direction notes
- TC: timecode or scene/chapter labels
- VISUALS: what the audience sees, written in present tense with motion

Every row must earn its place.
If a row does not introduce tension, progression, proof, or payoff, cut it.

## Timecode conventions by format

### Case study / award
Usually 1:30 to 2:00. Use named chapter labels when the story has distinct phases.

### Explainer / platform
Usually :60 to 1:30. Use tight timecode and feature progression.

### Promo / experiential
Scene-based or time-based depending on whether exact pacing matters.

### Teaser / cinematic
Scene labels when the film is mood-driven and timing is approximate.

## Visual column rules

Visuals must:
- Be present tense
- Contain motion and transformation, not static descriptions
- Describe transitions between sections explicitly
- Specify footage source when relevant: (stock), (archival), (AI-generated), (existing footage), (to-shoot)
- Separate SUPERs from scene description
- Distinguish hero shots from supporting b-roll
- Give motion graphics specific creative direction, not just "animated graphic"
- Never use "genuine," "authentic," or "real" as visual descriptors. These are adjectives hiding a missing scene.
- Never describe a category of footage ("food shots," "lifestyle moments"). Describe the specific shot.

## Asset reference rules

When the user attaches visual assets (images from PowerPoints, PDFs, or direct uploads), they are numbered [Asset 1], [Asset 2], etc.

Rules for referencing assets in the script:
- Reference each asset **at most once** in the client script. Pick the single best moment for each asset. Do not use the same [Asset N] tag in multiple rows.
- In the producer notes, refer to assets by description, not by [Asset N] tag. The asset thumbnails only render in the client script.
- If multiple sections could use the same asset, choose the most important moment and describe the other visuals without the asset tag.
- Not every asset needs to be referenced. Only use assets that genuinely fit the script. Do not force references.

## SUPER formatting

- Mark all SUPERs in bold: **SUPER:** "Text"
- Keep SUPERs to 1 to 5 words
- ALL CAPS, declarative, owned
- No hedging, no qualifiers
- Each super does one job: land a claim, name a player, punctuate a beat, or deliver a number
- Do not repeat what VO just said unless reinforcement is intentional

## SFX formatting

Italicize sound effects: *SFX: can crack > fizz*

## ALT VO rules

Use ALT VO only when:
- The opening could take two valid tonal paths
- The brief itself is ambiguous
- Leadership might want a more restrained option
- The line is conceptually right but tonally debatable

## Hard formatting rules

- NEVER use em dashes in VO or SUPERs. Use ellipses for pauses or separate sentences.
- Do not overwrite supers. A super that needs eight words is a headline that could not commit.
- Keep spacing readable. Do not cram.

## When to recommend a different deliverable structure

You may recommend:
- Master + cutdowns
- Master + localized swaps
- Modular city system
- Teaser + explainer companion piece

Do this when the brief is overloaded and one asset cannot solve the job cleanly. Name the recommendation and say why.


## Final validation before output

- Timecodes add up exactly to the target runtime
- No duplicate [Asset N] references
- No producer bleed into visuals
- No overwritten SUPERs
- Opening has tension
- Ending resolves, not just stops
