# Localization Systems

Many StudioNow scripts should not be written as one-off masters.
They should be written as systems.

## Use a modular system when:

- multiple cities or markets need versions
- the structure stays constant but details change
- the same film needs multiple stakeholders or audiences
- local pride matters
- runtime is too tight for full coverage in one master
- creative needs to scale without rewriting from scratch

## Always identify:

### Locked elements
These stay the same:
- overall concept engine
- edit rhythm
- main VO spine
- music approach
- brand moments
- closing lockup
- graphic system

### Variable elements
These change:
- city names
- local landmarks
- restaurant or retail partners
- market-specific stats
- event date/location cards
- local CTA
- localized hero shots

## Bracket syntax

Use clear brackets in the visual column:

[LOCALIZE: CITY NAME SUPER]
[LOCALIZE: skyline shot]
[LOCALIZE: hero food pairing]
[LOCALIZE: retail environment]
[LOCALIZE: event date/location card]

For VO:
[LOCALIZE: city name only]
or
ALT VO [CITY VERSION]

## Best practice for modular scripts

Write:
1. the master structure
2. what stays fixed
3. what swaps
4. sample city or market examples
5. any asset requirements per version

## Important rule

Do not pretend one :30 can meaningfully “show every city” if the list is long.
Instead recommend one of these:

### Option A: Route master
A fast, elegant overview with route graphics and abbreviated city callouts.

### Option B: Master + cutdowns
One national piece plus city-specific edits.

### Option C: Template-driven localized versions
One script system, many executions.

## What good localization feels like

Localization is not just dropping in a landmark.
It should reflect:
- local behavior
- local rhythm
- local food or retail truth
- local fan culture
- local emotional meaning

The audience should feel:
“This was made with my place in mind.”
